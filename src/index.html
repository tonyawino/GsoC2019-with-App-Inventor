  3. Proposed summer project

    IMPROVING THE LISTVIEW

    INTRODUCTION AND PROBLEM STATEMENT

    A ListView displays a vertically-scrollable collection of views, where each view is positioned immediately below the previous view in the list. The list items are inserted automatically to the list using an adapter. (“ListView  |  Android Developers,” n.d.)⁠

    Currently, App Inventor ListViews only accept Strings as their values. Only having Strings constraints users to wordy simplistic designs. (“User Interface Components - App Inventor for Android,” n.d.)⁠

    Given that adding different layouts to ListView is one of the most asked for improvements from users, I have recognized the need to add features to the ListView.(“Projects GSoC 2019 · mit-cml/appinventor-sources Wiki,” n.d.)⁠

    The solution currently being implemented to address this shortcoming is the use of jQuery Mobile ListView. However, this solution requires an understanding of HTML and the addition of extensions, which might be tough for beginners. (“App Inventor Tutorials and Examples: Listview | Pura Vida Apps,” n.d.)⁠

    The long-term solution is to review the ListView and possibly replace it with the newer RecyclerView. RecyclerViews should give the users much-needed flexibility. Also, RecyclerViews have a better performance as compared to the ListView as they reuse a fixed number of views as opposed to inflating all views for list items in the ListView.

    PROPOSED SOLUTIONS

    To improve users’ experience as the deal with lists, I propose to have two custom layouts for list items. On successful implementation of the ListView, a RecyclerView version of the solution will be developed. Features of the improved ListView will include:

    • Display lists of text only.
    • Display lists of images only.
    • Display lists of text and images.
    • Improved performance.

    POTENTIAL SOCIAL IMPACT

    Users will have the power to choose whether they want a list containing only text, only images, or images and text. This will also introduce features that have been requested by very many users over time.

    ACTION PLAN

    I plan to conduct the pilot on my own build server from my forked version of the official App Inventor sources. My server is only available to myself and will therefore not inconvenience other users as I work on the solution. The pilot will involve creating two custom layouts for list items, one containing only an ImageView, and the other containing an ImageView and a TextView. It will also involve creating a custom adapter to be called when a user requires a list of images or images and text. After successful implementation, the code will be sent for review.

    CRITERIA FOR SUCCESS

    In order to know that my project is successful, I will use the following metrics:

    • Successful completion of the Pilot phase of the project within one month.
    • Positive testimonials of impact and benefit from users of the new features.

  6. Challenge 2:Design challenge -- Improving App Inventor

    I settled on Spinner does not trigger AfterSelection on gone as it is mainly an Android runtime issue.

    The bug makes AfterSelection to be triggered only when a different item is selected from the spinner. Basically, selecting an item consecutively will only call AfterSelection the first time the item is selected. The issue occurs because Android’s Spinner onItemSelectedListener is only called when an object with an ID different from the currently selected object’s ID is selected. The issue can be solved by using a custom subclass of android.widget.Spinner instead of the android.widget.Spinner which is stored as variable view in the appinventor/components/src/com/google/appinventor/components/runtime/Spinner.java file. The appinventor/components/src/com/google/appinventor/components/runtime/util/HoneycombUtil.java file will also be modified to return the custom spinner when makeSpinner is called.

    Solving the problem would require overriding the setSelection method of the custom spinner as mentioned in this StackOverflow post . The setSelection method should call the spinner’s onItemSelectedListener each time it is called. The implementation would mean AfterSelection code is run whenever an item is selected. This could be useful in cases where a user expects feedback after selecting an item, but selecting the item twice gives no feedback, making the user doubtful of their actions.

    The custom spinner can be called MitSpinner and may look like:

    public class MitSpinner extends android.widget.Spinner {
    OnItemSelectedListener listener;
    public MitSpinner(Context context) {
    super(context);
    }
    public MitSpinner(Context context, int mode) {
    super(context, mode);
    }
    @Override
    public void setSelection(int position) {
    super.setSelection(position);
    if (listener != null)
    listener.onItemSelected(null, null, position, 0);
    }
    public void setOnItemSelectedEvenIfUnchangedListener(
    OnItemSelectedListener listener) {
    this.listener = listener;
    }
    }

    The component may be located on the directory containing the Spinner component.
